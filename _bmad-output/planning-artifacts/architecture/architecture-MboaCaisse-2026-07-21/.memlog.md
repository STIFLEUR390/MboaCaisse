- (decision) AD-1 Paradigme: Layered + Rich Domain. domain/ = comportement métier (pas structs anémiques). api/ = peau fine (parse→appel domain→sérialise). db/ = repositories. Pas d'hexagonal (boilerplate injustifié pour équipe 1-2 devs, infra stable: Axum+SQLite+USB).
- (decision) AD-2 Append-only financier: pattern système. Toute donnée financière (wallet_ledger, futures factures, commissions) utilise INSERT-only log. Wallet_ledger backup toutes les 5 min (P0). Décision: wallet-specific mais règle d'architecture qui s'étend.
- (decision) AD-3 Structure plate par couche: modules = fichiers dans api/, domain/, db/. Pas de sous-dossiers par capacité métier. Navigation plate pour équipe petite. Profondeur ajoutée quand >15 fichiers par dossier.
- (decision) AD-4 Payment et Wallet séparés mais liés. Wallet = comptes clients + ledger + identification téléphone. Payment = encaissement + multi-moyen + validation + écriture ledger. Payment → Wallet (appelle). Wallet → personne (île).
- (decision) AD-5 Print = service transverse (src/print.rs). Pas de route API, pas de domain. File d'attente asynchrone + writer ESC/POS. Appelé par Payment après validation. Ne bloque jamais la commande.
- (decision) AD-6 Reports = lecture seule, queries SQL directes dans api/reports.rs. Pas de domain/reports.rs. DTOs de présentation, pas d'agrégats.
- (decision) AD-7 Traits repository dans domain/, impl dans db/. api/ prend Arc<dyn XxxRepository>. Raison: frontière de compilation, pas mocking ou changement BDD.
- (decision) AD-8 Erreurs 3 couches sans fuite. db/ = DbError (interne). domain/ = DomainError enum (InsufficientBalance, ProductNotFound…). api/ = (StatusCode, Json<{error, code}>). Status code suffit (pas d'enveloppe {ok} générique). 200/201/400/401/403/422/500.
- (decision) AD-9 Cycle de vie: Tauri on_event ExitRequested → shutdown_tx → Axum graceful shutdown → backup BDD (timeout 5s). Mieux vaut perdre un backup qu'une corruption.
- (decision) AD-10 Stack: Rust edition 2021. Tokio/Axum versions flottantes (lockfile gèle). TanStack Query supprimé. useFetch() Nuxt + Pinia. rusqlite + r2d2. refinery pour migrations. Pas de WebSocket en V1.
- (decision) AD-11 Auth: JWT cookie (mboa_session, 24h, refresh silencieux si <1h restante). 4 rôles, permissions granulaires Vec<Permission>. Argon2. Seed auto admin + produits démo idempotent au 1er démarrage si BDD vide.
- (decision) AD-12 Config: tauri_plugin_store dans . Port (défaut 3000), mDNS hostname (mboacaisse), backup interval (24h), stock bas seuil (5), moyens paiement (['cash']). Pas de fichier YAML/TOML.
- (decision) AD-13 Graphe dépendances: Auth→personne. Catalog→personne. Order→Catalog+Wallet. Payment→Order+Wallet+Print. Wallet→personne (île). Stock→Catalog (lecture produits; conso stock P2). Kitchen→Order (lecture statut). Reports→lecture directe SQL toutes tables.
- (decision) AD-14 Kitchen display: polling HTTP toutes les 5s. Pas de WebSocket. useFetch() + setInterval dans kitchen.vue.
- (decision) AD-15 Migrations: refinery. SQL embarquées, table _schema_version auto-gérée. Runner au startup → si échec, log + exit. Pas de démarrage sans schéma validé.
- (decision) AD-16 Pool: r2d2-rusqlite. Mature, stable. deadpool-sqlite trop récent pour alpha. Switch si obsolescence.
- (decision) AD-17 Déploiement alpha: pas de staging, pas de Tauri updater. Binaire unique. Licence alpha pré-générée = entitlements avec flag alpha = logs DEBUG + diagnostics activés. Dev = poste dev. Prod = PC commerçant.
- (decision) AD-18 Logs: tracing + tracing-subscriber. Niveau INFO par défaut, DEBUG si licence alpha. Fichier mboacaisse.log dans . Pas de rolling en alpha.
